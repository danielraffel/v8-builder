# v8-builder — iOS (jitless) Lane Plan

**Status:** Scoping (exploratory). NOT a ship blocker.
**Date:** 2026-06-05
**Companion:** `planning/v8-builder-proposal.md` (§Phase 5), `planning/v8-builder-runbook.md` (P5).
**Premise:** Apple ships **JavaScriptCore** today and Pulp's iOS already uses it and works. The
only motivating use case for V8-on-iOS is **running three.js** under the same engine as desktop.
Treat this as a scoping exercise to size the work and surface the real unknowns — not a commitment.

Every claim below is grounded in the **real V8 15.1.27 checkout** synced on the Linux VM
(`admin@192.168.64.7:~/v8-builder/build/src/v8`), the existing `build-v8.py` / `seal/macho.py`
pipeline, and the V8 iOS docs. File/line citations are to that checkout.

---

## 1. Jitless — what it is, what turns off, and is interpreter-only enough for three.js

**The gn lever is `v8_enable_lite_mode`, which drives `v8_jitless`.** Confirmed in
`v8/BUILD.gn:439`:

```gn
v8_jitless = v8_enable_lite_mode          # BUILD.gn:439
```

and the Sparkplug derivation in `BUILD.gn:643-644`:

```gn
if (v8_enable_sparkplug == "") {
  v8_enable_sparkplug = !v8_jitless        # baseline JIT off when jitless
}
```

**On iOS this is automatic — we do not have to hand-set it.** `gni/v8.gni:167-174`:

```gn
# We use target_os == "ios" here because it isn't equivalent to is_ios
# (is_ios is based on host_os).
if (target_os == "ios") {
  if (target_platform == "iphoneos") {
    v8_enable_lite_mode = !use_blink       # v8.gni:170
  } else if (target_platform == "tvos") {
    v8_enable_lite_mode = true
    ...
  }
}
```

and `build/config/features.gni:41`:

```gn
use_blink = !is_ios                        # → false on iOS
```

So for an `iphoneos` (non-Blink) embed, `use_blink=false` ⇒ `v8_enable_lite_mode=true` ⇒
`v8_jitless=true`. The turbofan/wasm knobs then fall out of that in `gni/v8.gni:334-358`:

```gn
if (v8_enable_webassembly == "") { v8_enable_webassembly = !v8_enable_lite_mode }  # → false
if (v8_enable_turbofan == "")    { v8_enable_turbofan    = !v8_enable_lite_mode }  # → false
```

**What turns OFF in jitless/lite mode:**
- **TurboFan** (optimizing JIT) — `v8_enable_turbofan=false`.
- **Maglev** (mid-tier JIT) — gated on `v8_enable_turbofan` (`BUILD.gn:649-650`), so off.
- **Sparkplug** (baseline JIT) — `v8_enable_sparkplug = !v8_jitless` → false.
- **WebAssembly** — `v8_enable_webassembly=false` (WASM needs a JIT to emit code; the only
  jitless WASM path is the **drumbrake** interpreter, which V8 enables for **tvOS** but NOT
  iphoneos — `gni/v8.gni:172-181`). So on iPhone/iPad: **no `WebAssembly` global**.
- `V8_LITE_MODE` and `V8_JITLESS` defines are set (`BUILD.gn:1286-1288`, `1463-1465`).

**What stays ON:** the **Ignition** bytecode interpreter — the full ECMAScript language runs,
just interpreted. This is the same execution tier JSC's iOS interpreter and QuickJS use.

**Is interpreter-only enough for three.js? — Yes, functionally; the risk is performance, not
correctness.** three.js is pure ES (classes, typed arrays, `Math`, closures). None of it needs
JIT or WASM. So a jitless V8 *runs* three.js. The open question is whether interpreted V8 is fast
enough to drive `THREE.WebGPURenderer` at interactive frame rates on an A-series CPU — and here
the honest answer is **JSC on iOS already does this with its own interpreter**, so the value of
swapping in jitless V8 is "same engine as desktop," not "faster." That is exactly why this lane is
exploratory (see §7).

**WebAssembly interaction (the load-bearing one):** `v8_enable_webassembly` MUST be off on iOS
device — it is implied by lite mode and is also the correct App Store posture (no JIT). The
`three.webgpu.js` bundle Pulp ships is plain JS and does not require WASM, so this is acceptable.
If a future Pulp feature needs WASM on iOS, jitless V8 cannot provide it (and neither can App-Store
JSC) — that is a platform ceiling, not a build bug.

## 2. gn args for iOS — what changes vs `mac_gn_args`, and the static-vs-shared question

### 2a. The arg delta

Starting from `common_gn_args()` (the sealed-shared baseline in `build-v8.py:54-85`), an iOS
device cell is `mac_gn_args` with these substitutions/additions. Grounded in the V8 iOS doc
(`v8.dev/docs/cross-compile-ios`) and the checkout:

```gn
target_os   = "ios"
target_cpu  = "arm64"            # arm64 device; "arm64" or "x64" for simulator (see 2b)
target_environment = "device"    # REQUIRED, not defaulted — mobile_config.gni:10 ("")
target_platform = "iphoneos"     # default (mobile_config.gni:16); selects the lite-mode branch
ios_deployment_target = "14.0"   # pick Pulp's floor; V8 default is "18.0" (use_blink=false)
                                 #   build/config/ios/ios_sdk_overrides.gni:16
is_component_build = false        # iOS doc + monolith both require non-component
use_custom_libcxx = false         # iOS doc: use the platform libc++ (matches Skia/Dawn-on-iOS)
v8_monolithic = true              # same as desktop
v8_monolithic_for_shared_library = true   # keep — TLS reloc fix, harmless on Mach-O
v8_use_external_startup_data = false      # embed the snapshot (no loose .bin in the bundle)
v8_enable_pointer_compression = false     # iOS doc sets false; matches our desktop D3 OFF posture
# v8_enable_lite_mode / v8_jitless / turbofan / wasm: DO NOT SET — derived automatically
#   from target_os=="ios" + target_platform=="iphoneos" (see §1). Setting them by hand is
#   redundant and risks fighting the gni resolution.
```

**i18n / Intl — the one deliberate divergence from the desktop flagship.** Desktop builds Intl
**ON** (`v8_enable_i18n_support=true`, D2). The V8 iOS doc sets `v8_enable_i18n_support=false`.
Recommendation for the exploratory lane: **keep Intl OFF on iOS first** (smaller binary, no ICU to
seal, faster bring-up, and Pulp itself uses no `Intl` — grep finding in the proposal §D2). If a
general-audience iOS artifact later wants Intl, the seal handles it the same way as desktop (ICU
stays internal behind the export boundary) — but prove the lane jitless+Intl-off first.

**Deployment target:** the checkout defaults `ios_deployment_target` to **18.0** when
`use_blink=false` (`ios_sdk_overrides.gni:13-17`). That is almost certainly higher than Pulp's iOS
floor — set it explicitly to whatever Pulp's `ios` skill / Package.swift targets (commonly 14.0 or
15.0). Record the chosen value in `manifest.json`.

### 2b. Device vs simulator (`target_environment`)

`target_environment` is **mandatory** — `mobile_config.gni:10` declares it `""` and
`mobile_config.gni:96-105` asserts it is one of `device|simulator|catalyst`. So the iOS lane is
**two cells minimum**, mirroring how skia-builder ships iOS (see §5):

| Cell | target_os | target_cpu | target_environment | Runs on |
|------|-----------|------------|--------------------|---------|
| iOS device | ios | arm64 | device | real iPhone/iPad (needs codesign) |
| iOS simulator (arm64) | ios | arm64 | simulator | Apple-silicon Sim |
| iOS simulator (x86_64) | ios | x64 | simulator | Intel Sim / Rosetta |

For CI/local validation without a device, the **arm64 simulator** cell is the cheap path (no
codesigning, runs on the Mac Studio's Sim). The x86_64 simulator cell is optional (only needed for
Intel-host Sim; given the all-Apple-silicon fleet, low priority).

### 2c. Static `.a` vs shared `.dylib` vs embedded `.framework` — the architectural fork

This is where iOS diverges hardest from the desktop flagship and is the **#1 design decision**.

- The V8 iOS doc builds **`v8_monolith` static (`libv8_monolith.a`)** and links it straight into
  the app — that is the blessed iOS shape.
- Our desktop flagship is a **sealed shared library** built via the injected `v8_shared_library`
  gn target (`build-v8.py:184-236`). But `gni/v8.gni:553-571` shows `v8_shared_library` is just a
  thin wrapper over gn's `shared_library()` — on iOS that emits a **bare `.dylib`**, NOT an
  embeddable framework.
- **iOS app-store / loader rules:** iOS apps may embed dynamic libraries, but they must ship as a
  signed **`.framework`** inside `App.app/Frameworks/` (a loose `.dylib` is rejected/awkward; the
  dynamic linker and codesign expect the framework bundle layout). V8's gn tree HAS an
  `ios_framework_bundle` template (`build/config/ios/rules.gni:803`) — but our injected
  `v8_shared_library` target does not use it.

**Recommendation: ship iOS as a STATIC `libv8_monolith.a` for the exploratory lane** (matches the
V8 iOS doc, sidesteps framework bundling and codesigning entirely), and treat "sealed iOS
framework" as a later step only if a general-audience iOS artifact demands it. This trades away
the uniform export-table seal (see §4) but is the lowest-risk path to "three.js runs on a jitless
V8 in the Sim."

## 3. Build host — confirmed mac, disk budget

**iOS cross-compiles on the Mac, exactly like the mac lane.** It uses Apple's clang + the iOS SDK
(via `build/config/ios/ios_sdk.gni`), driven by the same `depot_tools`/`gn`/`ninja` flow already in
`build-v8.py`. This is fundamentally different from the Linux/Windows cells (which need V8's
bundled x86_64-Linux-host clang-23 / Rust and therefore CI or Rosetta-for-Linux): **iOS has no such
constraint** — it builds with the host Xcode toolchain on any Apple-silicon Mac. That is why iOS
**parallelizes cleanly with the Android lane** (Android = Linux-VM/NDK work; iOS = mac-host work;
no shared build farm).

**Disk:** the proposal budgets ~50 GB per V8 workspace. The good news is iOS **reuses the same
`build/src/v8` checkout** already synced — only a new `out/ios-<env>-<arch>` gn output dir is added
(each output tree is a few GB). No second 50 GB sync. Plan for ~5-10 GB additional per iOS cell on
top of the existing checkout.

## 4. The seal — Mach-O, but static-`.a` changes how (and whether) it applies

**The seal mechanism is arch-independent and already Mach-O.** `seal/macho.py` works on symbol
patterns (`__ZN2v8*`, `__ZN6cppgc*`, …) via `-exported_symbols_list` + an `nm -gU` audit
(`seal/macho.py:24-27, 43-54, 58-77`). Nothing in it cares about arm64-vs-x86_64 or
device-vs-simulator — an iOS arm64 dylib seals identically to a macOS arm64 dylib. Bitcode is
**dead** (Xcode 14+ dropped it), so there is no bitcode-archive concern.

**BUT the static-`.a` decision (§2c) breaks the seal model.** The export-table seal only exists for
a **dynamic** image:
- A `.dylib`/`.framework` has a dynamic **export table** — `-exported_symbols_list` hides
  ICU/zlib/Abseil from it by construction (the whole point of the desktop flagship).
- A static **`libv8_monolith.a`** is just an archive of `.o` files with **no export table at
  all**. Every symbol in it is a regular global `.o` symbol. There is nothing for
  `-exported_symbols_list` to filter. So a static iOS lib is **NOT sealed the way the desktop
  shared libs are.**

**Consequences and options for a static iOS `.a`:**
1. **Accept "unsealed static" for the exploratory lane (recommended).** On iOS there is **no
   Dawn-vs-V8 Abseil collision to seal against the way desktop has** — but verify this against
   Pulp's actual iOS GPU stack. Pulp's iOS uses **Dawn/Metal** (per the task), and Dawn bundles
   Abseil, so the *same* P0.3a ODR hazard (V8's Abseil vs Dawn's Abseil) can recur at the final app
   link. If it does, a static V8 cannot hide its Abseil via an export table.
2. **Seal the static `.a` at the object level (the hard path).** This is exactly the
   "single-object internalization" the proposal originally feared for Windows static `.lib`
   (`v8-builder-proposal.md` §6, §6b): `ld -r` the whole monolith into one relocatable `.o`, then
   localize everything that isn't `v8::`/`cppgc::`. On Mach-O this is `ld -r
   -exported_symbols_list public.txt`. It is a real technique but **unproven in our context** and
   is the genuine new work iOS would add over the desktop seal.
3. **Build the iOS lane as a sealed `.framework`** (re-introduce the dynamic export boundary via
   `ios_framework_bundle`, `rules.gni:803`). This restores the clean desktop-style seal, at the
   cost of framework bundling + codesigning + the loader-path work.

**Decision for scoping:** start with option 1 (unsealed static), and **make the validator's first
job determine whether the V8↔Dawn Abseil ODR actually fires on iOS** (link a fixture that pulls
both). If it fires, escalate to option 2 or 3. The honest unknown is which.

## 5. Skia/Dawn pairing — skia-builder already ships iOS

`gh release view chrome/m150 --repo danielraffel/skia-builder` lists the iOS coexistence targets:

```
skia-build-ios-device-arm64-gpu-release.zip
skia-build-ios-simulator-arm64-x86_64-gpu-release.zip
Skia.xcframework.zip
```

(plus `visionos-device`/`visionos-simulator` for later.) So the coexistence partner already exists,
in exactly the two cells we need:

- **`ios-simulator-arm64-x86_64`** (a fat/xcframework slice) pairs with our simulator cells.
- **`Skia.xcframework.zip`** is the bundled multi-platform form — the most natural thing to link a

with the desktop Skia. The validator (§6) downloads the matching `skia-build-ios-*` zip exactly as
`build-v8.yml:108-118` does for desktop.

## 6. Validation — no-skip identity check on iOS, minimal harness

There is **no CLI on iOS** — you cannot `./v8_skia_validate` the way the desktop lane does
(`validate/identity_main.cpp`). The minimal harness is a tiny **Simulator app** (or XCTest bundle)
that runs the *same* identity logic and reports via stdout/exit.

**Reuse the existing identity contract verbatim.** `validate/identity_main.cpp` already does the
no-skip-pass identity check: `v8::V8::Initialize()` (aborts on an Abseil ODR), eval `20+22`, assert
`== 42`, assert `v8::V8::GetVersion() == EXPECTED_V8_VERSION`, and force the collision partners
(Dawn/Skia ICU/Abseil). The iOS harness should call **exactly this code** from an app's
`main()`/`application(_:didFinishLaunching:)`, plus one jitless-specific assertion:
- assert WASM is absent (`typeof WebAssembly === 'undefined'`) to prove the jitless build shape;
- optionally eval a tiny three.js-shaped snippet (typed arrays + matrix math) to prove the
  interpreter handles the real workload.

**Minimal harness shape (scope):**
1. A gn `ios_app_bundle` (or `xctest`) target injected into `BUILD.gn` — mirror the existing
   `inject_win_validator()` pattern (`build-v8.py:427-456`), which already builds an identity
   validator **as a gn target** so it inherits V8's exact toolchain/defines (`:external_config`).
   The iOS analog deps `:v8_sealed_shared` (or `:v8_monolith` if static), `configs +=
   [":external_config"]` so `V8_JITLESS`/compression defines match.
2. **Run it headless via `xcrun simctl`:** `xcrun simctl boot <udid>` → `install` → `launch
   --console-pty` the app, scrape the `PULP_ENGINE_IDENTITY_BEGIN…END` block from the console,
   assert exit/identity. This is the iOS equivalent of the desktop "run the binary" step. The
   arm64 Simulator on the Mac Studio runs the arm64-sim cell natively (no codesigning).
3. Device-cell validation needs a provisioned device + codesign — defer; the simulator cell is the
   gating proof for "jitless V8 runs three.js-shaped JS."

**Audio-etiquette caveat (Pulp CLAUDE.md):** booting an iOS Simulator app can open a virtual
coreaudio device and emit sound. The three.js demo is silent, but per the interim contract: announce
before launching the Sim, cap the run (terminate after the identity block fires or ~90s), and tear
down (`xcrun simctl terminate` + `shutdown`). For a pure JS-identity harness with no audio bus this
is low-risk, but follow the contract.

## 7. Effort, risks, and why it's lower priority

**Effort: HIGHER than Android, and meaningfully so.** Android reuses the desktop sealed-shared
recipe almost wholesale (NDK cross of a `.so`, same export-table seal, `readelf` audit). iOS adds
**three genuinely new pieces** the desktop/Android lanes never solved:
1. **Jitless build shape** — new gn posture (auto-derived, but the *consumer* must compile with
   `V8_JITLESS` and accept no-WASM; choc/Pulp's V8 TU has never been built jitless).
2. **Static-`.a` sealing** (§4) — the desktop seal is an export table; a static iOS lib has none,
   so either accept unsealed (and hope no Dawn-Abseil ODR) or do object-level internalization
   (the unsolved "single-object" technique the proposal feared for Windows static `.lib`), or
   build a sealed `.framework` (bundling + codesigning).
3. **Simulator harness** — no CLI; an `ios_app_bundle` + `xcrun simctl` console-scrape, vs the
   desktop's plain `./binary`.

Rough sizing: desktop mac arm64 took the bulk of Phase 0/1/2. iOS is **~1.5-2x a desktop cell** —
not because compiling is hard (it cross-compiles on the Mac with the host toolchain, §3) but
because of the seal-model fork (§4) and the harness (§6). The two unknowns (Abseil-on-iOS, sim
harness) could each eat a day.

**Parallelizes with Android — yes.** iOS is **mac-host** work (Apple clang + iOS SDK, §3); Android
is **Linux-VM/NDK** work. They share the symbol-level seal *policy* but not the build farm, the
toolchain, or the validation host. They can proceed in parallel without contention. iOS even reuses
the already-synced `build/src/v8` checkout (§3) — no second 50 GB sync.

**Risks:**
- **V8↔Dawn Abseil ODR on iOS** (the big one) — a static V8 can't seal Abseil; unknown whether it
  fires with Pulp's iOS Dawn/Metal stack. The validator must probe this first.
- **Interpreted-V8 performance** for three.js on A-series — may be slower than desktop's JIT'd V8;
  may be *no better than the JSC Pulp already uses*. This is the crux of "why bother."
- **No WASM on iOS device** — a permanent ceiling of jitless; acceptable for current three.js, not
  for any future WASM-dependent feature.
- **Deployment-target / SDK drift** — V8's checkout defaults `ios_deployment_target=18.0`; must be
  pinned to Pulp's floor.

**Why it's lower priority — stated plainly:** **Apple ships JavaScriptCore, Pulp's iOS already uses
it, and it works.** A jitless V8 on iOS buys "same engine as desktop" and nothing else measurable —
it is interpreter-vs-interpreter against JSC, with no JIT or WASM advantage available on the
platform. So this lane is **exploratory**: worth scoping and a sim-only proof to know it's
*possible*, but it should land **after all desktop lanes are green and PRs merged** (the runbook
already sequences it at P5, after desktop + m150), and it is explicitly **not a ship blocker**.

## Appendix — concrete gn snippet for an iOS simulator (arm64) cell

A `ios_gn_args("arm64", env="simulator")` helper, modeled on `mac_gn_args` (`build-v8.py:93-101`):

```python
def ios_gn_args(arch, env="device"):
    # env in {"device","simulator"}; target_cpu arm64 (device|sim) or x64 (intel sim)
    return common_gn_args() + [
        f'target_cpu="{gn_cpu(arch)}"',
        'target_os="ios"',
        f'target_environment="{env}"',     # REQUIRED (mobile_config.gni asserts)
        'target_platform="iphoneos"',      # selects the lite-mode (jitless) branch
        'ios_deployment_target="14.0"',    # pin to Pulp's floor (V8 default is 18.0)
        'is_component_build=false',
        'use_custom_libcxx=false',
        'v8_enable_i18n_support=false',     # Intl OFF first (no ICU to seal on iOS lane)
        # v8_enable_lite_mode / v8_jitless / turbofan / wasm: auto-derived — do NOT set.
        # v8_enable_pointer_compression stays false (common_gn_args already sets it).
    ]
```

The `gn_cpu` map (`build-v8.py:89-90`) already handles `x86_64→x64`; iOS adds no new arch labels.
A static-`.a` lane would skip the injected `v8_sealed_shared` target entirely and ship
`out/ios-<env>-<arch>/obj/libv8_monolith.a` directly (no Mach-O export seal — see §4).

## Review addendum — RepoPrompt + Codex (2026-06-05, both concur)

Two independent reviews converged hard. The big redirect + corrections:

1. **KEY REDIRECT: go a sealed dynamic FRAMEWORK / XCFramework, NOT static `.a`.** Both
   reviewers: a static `libv8_monolith.a` has no dynamic export table, so the seal can't
   apply, and V8's Abseil/ICU/zlib globals merge into the final app link — recreating the
   exact V8↔Dawn Abseil ODR abort this project exists to prevent (Pulp iOS uses Dawn/Metal).
   Wrapping a *shared* lib in `ios_framework_bundle` with `-exported_symbols_list` (+ `@rpath`
   install name, codesign, device/sim slices) **keeps the existing Mach-O seal model** — the
   natural escape. Object-level `ld -r` internalization is fragile; avoid. Static-unsealed is
   an **experiment**, not a sealed v8-builder lane.
2. **DO FIRST (the gate):** probe whether the Abseil ODR actually fires on iOS V8 + Dawn/Metal.
   That single result decides whether *any* iOS lane is viable — before build investment.
3. **Intl bug:** the plan appends `v8_enable_i18n_support=false`, but `common_gn_args()`
   hard-sets it `true` → a GN **duplicate-assignment**. Must **filter it out of the common
   list first** (the `win_gn_args` pointer-compression pattern). (Also reconsider whether we
   even want Intl off — keep it ON for desktop parity unless size is critical.)
4. `v8_monolithic_for_shared_library=true` (from `common`) is **not** needed for a static `.a`
   — but **is** correct for the framework/dylib route. Don't justify it as a static-lane TLS fix.
5. **Validation:** `simctl launch --console-pty` is **not** a CLI exit-code contract — the app
   must signal pass/fail via a log marker / result file; scrape the identity block + terminate
   explicitly (audio-etiquette teardown). The dep is `:v8_monolith` (static) vs the
   framework/`:v8_sealed_shared` (dynamic) — pick one and be consistent. And **force the
   Dawn/Abseil collision explicitly** — `smoke_gpu.cpp` only forces Skia ICU today.
6. If iOS ships a sealed framework, **harden `macho.py`'s audit to an allow-list** (it's
   deny-check-only today, weaker than `elf.py`). Manifest fields to add: `jitless`,
   `wasm=false`, `i18n`, `deployment_target`, `shared|static`, `sealed`.

**Build-environment gotcha — flaky-network gsutil wedge (worked around):** on an
unstable network, depot_tools' `gclient sync` hooks that use **gsutil's boto `gs://`
transport wedge every download in CLOSE_WAIT** — even a single serial `gsutil cp` —
and never recover, stalling the sync indefinitely. The wedge is transport-specific:
plain **HTTPS** to `storage.googleapis.com` / `commondatastorage.googleapis.com`
and **CIPD's** HTTPS transport both work fine on the same network. Recovery recipe
(used to unblock this gate without a full sync):
1. `tools/clang/scripts/update.py` and `tools/rust/update_rust.py` fetch the
   clang + Rust toolchains over HTTPS (no gsutil).
2. `cipd ensure -root buildtools/mac -ensure-file <gn.ensure>` fetches the pinned
   `gn` binary (gn_version from DEPS) over CIPD/HTTPS.
3. `tools/prefetch_gcs_https.py` fetches the remaining
   `download_from_google_storage` (.sha1-named) objects over HTTPS and verifies the
   sha1, so the gsutil hooks find them present and skip. (Auth-gated dev-only
   buckets — gcmole/jsfunfuzz/bazel — 403 anonymously and are skipped; not needed
   for the framework build.)
4. Reconstruct `build/config/gclient_args.gni` from the DEPS `gclient_gn_args` list
   (`android_ndk_version`, `checkout_src_internal=false`) — gclient normally writes
   it during the sync hook phase that the wedge blocked.
With those in place, `gn gen` + `ninja v8_sealed_shared` build the sealed framework
directly. This is a build-host network condition, not a lane bug; a healthy network
or CI runner does the normal full `gclient sync`.
